## Summary

The live repo is still npm-based. `package-lock.json` is present, `pnpm-lock.yaml` and `.npmrc` are absent, `package.json` has no `packageManager` field, `Makefile`/`lefthook.yml` still use `npx`, and `.github/workflows/ci.yml` still uses `npm ci`, `cache: 'npm'`, and `npm audit`. The historical quick-task artifacts are not the source of truth.

The safest migration scope is **not** “delete `package-lock.json` and run `pnpm install`”. In two disposable clone rehearsals on 2026-03-27, `pnpm import && pnpm install` preserved the current resolved graph and worked without any hoisting config, while a plain `pnpm install` upgraded direct dependencies inside existing semver ranges (`@adonisjs/core` `7.1.1 -> 7.2.0`, `@adonisjs/assembler` `8.2.0 -> 8.3.0`). If the goal is a tooling swap and not a dependency refresh, `pnpm import` is the correct first step.

The old assumption that AdonisJS v7 here requires `shamefully-hoist=true` is not supported by current evidence. In a clean clone with no `.npmrc`, `pnpm import`, `pnpm install`, `pnpm install --frozen-lockfile`, `pnpm exec tsc --noEmit`, and `node ace build` all succeeded. `pnpm` 10 did warn about blocked build scripts for `@swc/core` and `lefthook`, but those warnings did not break install, typecheck, or build. Treat build-script approval as a watchpoint, not an automatic config change.

**Primary recommendation:** Plan this as a lockfile-preserving migration: add an exact `packageManager` pin, generate `pnpm-lock.yaml` with `pnpm import`, switch `npx`/CI install-audit paths to `pnpm exec` and `pnpm install --frozen-lockfile`, and do **not** add hoisting config unless a real failure proves it is needed.

## Architecture Patterns

### Recommended Project Structure

```text
project-root/
├── package.json              # add exact "packageManager" pin; keep scripts package-manager-agnostic
├── pnpm-lock.yaml            # canonical lockfile generated from package-lock via pnpm import
├── Makefile                  # replace npx-only targets with pnpm exec targets
├── lefthook.yml              # replace npx-only hook commands with pnpm exec
└── .github/workflows/ci.yml  # install/cache/audit with pnpm in all jobs
```

**Do not add by default:** `.npmrc`, `pnpm-workspace.yaml`, or hoisting config. No current evidence requires them.

### Pattern 1: Lockfile-Preserving Migration

**What:** Generate `pnpm-lock.yaml` from the current `package-lock.json`, then reinstall from a clean `node_modules`.

**When to use:** Migrating a live repo where dependency upgrades are not the goal.

**Example:**

```bash
# Source: https://pnpm.io/cli/import
pnpm import
rm -rf node_modules
pnpm install --frozen-lockfile
# delete package-lock.json only after this passes
```

### Pattern 2: Pin pnpm Centrally

**What:** Add an exact `packageManager` pin to `package.json` and let local tooling/CI consume it.

**When to use:** You want one authoritative pnpm version instead of “whatever happens to be installed”.

**Example:**

```json
// Source: https://pnpm.io/installation
{
  "packageManager": "pnpm@10.33.0"
}
```

### Pattern 3: Run Project Binaries with `pnpm exec`

**What:** Replace `npx` in `Makefile` and `lefthook.yml` with `pnpm exec`. Keep `node ace ...` commands unchanged.

**When to use:** The command targets a dependency already declared in this repo.

**Example:**

```makefile
# Source: https://pnpm.io/cli/exec
lint:
	pnpm exec eslint . --max-warnings 0

typecheck:
	pnpm exec tsc --noEmit
```

### Pattern 4: Official GitHub Actions pnpm Setup

**What:** Install pnpm with `pnpm/action-setup`, install Node with `actions/setup-node`, cache pnpm store, and use `pnpm install --frozen-lockfile`.

**When to use:** GitHub Actions CI for this repo.

**Example:**

```yaml
# Source: https://pnpm.io/continuous-integration
# Source: https://github.com/pnpm/action-setup
# Source: https://github.com/actions/setup-node
- uses: actions/checkout@v4
- uses: pnpm/action-setup@v5
- uses: actions/setup-node@v6
  with:
    node-version: 24
    cache: pnpm
- run: pnpm install --frozen-lockfile
```

If `package.json` has `"packageManager": "pnpm@10.33.0"`, `pnpm/action-setup` can read that pin and does not need a separate `version:` input.

### Recommended Plan Decomposition

**Plan A: Lockfile + pin + clean install**

- Add `"packageManager": "pnpm@10.33.0"` to `package.json`
- Run `pnpm import` to create `pnpm-lock.yaml`
- Reinstall from clean `node_modules` with `pnpm install --frozen-lockfile`
- Delete `package-lock.json` only after the frozen install passes

**Plan B: Command surface migration**

- Update `Makefile` `lint`, `lint-fix`, `typecheck`, and add `audit`
- Update `lefthook.yml` commands from `npx` to `pnpm exec`
- Keep existing `node ace` commands as-is

**Plan C: CI migration + final verification**

- Update `.github/workflows/ci.yml` to use `pnpm/action-setup@v5`
- Update `.github/workflows/ci.yml` to use `actions/setup-node@v6` with `cache: 'pnpm'`
- Replace `npm ci` with `pnpm install --frozen-lockfile`
- Replace `npm audit` with `make audit` (backed by `pnpm audit --audit-level=high`)
- Validate in a clean checkout / CI under Node 24

### Anti-Patterns to Avoid

- **Deleting `package-lock.json` before `pnpm import`:** This causes avoidable version drift.
- **Adding `shamefullyHoist=true` up front:** Current repo evidence does not justify it.
- **Using `pnpm dlx` for checked-in tools:** `pnpm exec` is the correct local-bin command.
- **Keeping both lockfiles:** AdonisJS tooling detects package manager from lockfiles; mixed lockfiles create ambiguity.
- **Using two pnpm cache mechanisms in CI:** Pick one cache path, not both.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| npm -> pnpm lockfile conversion | Manual lockfile rewrites or “delete and reinstall” | `pnpm import` | Preserves the current resolved graph |
| Project-local tool execution | Ad hoc `.bin` paths or `pnpm dlx` | `pnpm exec` | Uses declared dependencies without remote fetches |
| GitHub Actions pnpm bootstrap | Shell install scripts in workflow | `pnpm/action-setup` + `actions/setup-node` | Official, version-aware, cache-aware |
| Broad hoisting workaround | Blanket `shamefullyHoist=true` | Default isolated layout; targeted `publicHoistPattern` only if needed | Minimizes phantom dependency exposure |
| pnpm 10 build-script allowlisting | Custom shell hacks | `pnpm approve-builds` / explicit pnpm config only if a package truly breaks | Matches pnpm’s current security model |

**Key insight:** This should be a **package-manager migration**, not a dependency-upgrade phase and not a node_modules-layout rewrite phase.

## Common Pitfalls

### Pitfall 1: Version Drift by Skipping `pnpm import`

**What goes wrong:** A direct `pnpm install` upgrades packages inside the existing semver ranges.

**Why it happens:** `package.json` uses `^` and `~`, and deleting the npm lock discards the frozen graph.

**How to avoid:** Run `pnpm import` first, keep `package-lock.json` until `pnpm install --frozen-lockfile` succeeds, then remove it.

**Warning signs:** Direct dependencies jump from `@adonisjs/core@7.1.1` to `7.2.0` and `@adonisjs/assembler@8.2.0` to `8.3.0`.

### Pitfall 2: Unnecessary Root Hoisting

**What goes wrong:** Adding `shamefullyHoist` makes phantom dependencies available at the root and hides real resolution bugs.

**Why it happens:** Older migration advice treats hoisting as a generic fix.

**How to avoid:** Start with pnpm defaults. Only add targeted `publicHoistPattern` if a specific tool fails under isolated linking.

**Warning signs:** Hoisting config gets added before any reproducible module-resolution failure exists.

### Pitfall 3: CI Only Half-Migrated

**What goes wrong:** Local dev uses pnpm but CI still caches npm or runs `npm ci` / `npm audit`.

**Why it happens:** The migration updates lockfiles and hooks but misses the workflow file.

**How to avoid:** Update every job in `.github/workflows/ci.yml`, especially `security`.

**Warning signs:** `cache: 'npm'`, `npm ci`, or `npm audit` remain in the workflow.

### Pitfall 4: Confusing `pnpm exec` with `pnpm dlx`

**What goes wrong:** Hooks or Make targets fetch tools from the registry instead of using the repo’s declared dependencies.

**Why it happens:** `dlx` looks similar to `npx`.

**How to avoid:** Use `pnpm exec` for `eslint`, `prettier`, `tsc`, and `commitlint`.

**Warning signs:** Hook commands mention `dlx` even though the same tool is already in `package.json`.

### Pitfall 5: Local Validation on the Wrong Machine State

**What goes wrong:** A developer validates on Node 22 or from a polluted workspace and blames pnpm for unrelated failures.

**Why it happens:** This repo requires Node 24, and the current workspace has untracked `.claude/worktrees/**` directories that `eslint .` recurses into.

**How to avoid:** Validate in CI, Docker, or a clean checkout under Node 24. Treat `.claude/worktrees/**` lint explosions as local-environment noise unless the repo intentionally tracks those files.

**Warning signs:** Engine warnings or thousands of ESLint parser errors under `.claude/worktrees/**`.

### Pitfall 6: Misreading pnpm 10 Build-Script Warnings

**What goes wrong:** Install logs warn about blocked build scripts for `@swc/core` and `lefthook`, and the phase adds extra config immediately.

**Why it happens:** pnpm 10 defaults are stricter about dependency build scripts.

**How to avoid:** Verify whether typecheck/build/tests actually fail first. Add approvals only if required.

**Warning signs:** Warning appears, but `pnpm exec tsc --noEmit` and `node ace build` still pass.

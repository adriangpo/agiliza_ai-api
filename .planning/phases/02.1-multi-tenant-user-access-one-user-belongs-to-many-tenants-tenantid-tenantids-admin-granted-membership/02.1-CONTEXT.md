# Phase 02.1: Multi-Tenant User Access тАФ Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Rework the user-tenant relationship from 1:1 to M:N. A single identity can access multiple tenants,
with manager access granted by a platform admin. Citizens remain global (no tenant affiliation).
The phase also adds tenant geographic boundaries (PostGIS polygons) to enable GPS-based tenant
resolution in Phase 3.

This phase edits the original migrations in-place (dev environment тАФ no production data, no conflict
risk). No "fix" migrations are needed.

**What this phase delivers:**
- Global user identity (no tenant_id on users)
- `tenant_memberships` join table (manager тЖФ tenant links)
- `is_superuser` flag for platform admins
- `boundary` polygon on tenants table (PostGIS)
- Updated auth flow (no X-Tenant-ID at register/login)
- Updated TenantMiddleware (validates manager memberships)
- Argon2 password hashing (replaces scrypt)

**What this phase does NOT deliver:**
- Admin endpoints to create managers or grant memberships тЖТ Phase 02.2
- GPS-based automatic tenant resolution at report submission тЖТ Phase 03
- Citizens reading reports across tenants тЖТ Phase 03

</domain>

<decisions>
## Implementation Decisions

### Identity Architecture

- **D-01:** `users` table becomes global тАФ remove `tenant_id` FK, remove `role` column. Email is
  globally unique (system-wide). Add `is_superuser: boolean NOT NULL DEFAULT false`. Edit the
  original `002_auth_users.ts` migration in-place.

- **D-02:** Create `tenant_memberships` table:
  ```
  tenant_memberships
    id            bigint serial PK
    user_id       bigint FK тЖТ users.id ON DELETE CASCADE
    tenant_id     uuid FK тЖТ tenants.id ON DELETE CASCADE
    granted_by    bigint FK тЖТ users.id (platform admin who granted access)
    granted_at    timestamp NOT NULL
    UNIQUE(user_id, tenant_id)
  ```
  No `role` column тАФ presence in this table IS the manager role for that tenant.

- **D-03:** Remove RLS from the `users` table entirely. Users are globally accessible by the app
  role. Tenant isolation (RLS) lives on data tables (reports, incidents, etc.) which carry
  `tenant_id` determined at write time.

- **D-04:** Add `boundary` geography(POLYGON, 4326) column to `tenants` table (nullable тАФ not all
  tenants will have a boundary set immediately). This column is the foundation for GPS-based tenant
  auto-detection in Phase 3. Edit `001_foundation_tenants.ts` in-place.

### Authorization Model

> **CRITICAL тАФ document this in every policy and middleware file.**
>
> The authorization hierarchy is:
> 1. `user.isSuperuser === true` тЖТ platform admin, bypasses all tenant restrictions
> 2. `tenant_memberships` row exists for (user.id, requested tenantId) тЖТ manager for that tenant
> 3. Otherwise тЖТ citizen (no tenant context, operates globally)
>
> There is NO explicit `role` column anywhere. Authorization is determined by DB state,
> not by a stored role string. Code must make this explicit via comments.

- **D-05:** `is_superuser` boolean column on `users` (default false). Platform admins are created
  directly (seeders or Phase 02.2 admin endpoint). Bouncer checks `user.isSuperuser` first before
  any tenant restriction.

- **D-06:** All members in `tenant_memberships` are equal тАФ no permission tiers within a tenant in
  v1.0. If auditor/read-only roles are needed later, add a `role` column to `tenant_memberships`
  then. For now, "has membership = full manager access."

- **D-07:** Token abilities are neutral at login time тАФ all tokens carry:
  `['reports:write', 'reports:read', 'profile:manage']`.
  Manager capabilities are NOT encoded in the token. They come from loading the `tenant_memberships`
  row when `X-Tenant-ID` is present. Bouncer reads `ctx.membership` (loaded by TenantMiddleware).

### Tenant Context in Requests

- **D-08:** Managers pass `X-Tenant-ID` header on every authenticated request where they need
  tenant context (dashboard, moderation, etc.). `TenantMiddleware` validates:
  1. `X-Tenant-ID` header is present and is a valid UUID
  2. A row exists in `tenant_memberships` for `(user.id, tenantId)` (or user is superuser)
  3. If valid: set `app.tenant_id` via `set_config`, attach `ctx.tenantId` and `ctx.membership`
  4. If invalid/missing: return 403 Forbidden

- **D-09:** Citizens making authenticated requests do NOT use `TenantMiddleware`. Citizen routes
  have no tenant scoping at the auth layer тАФ tenant isolation for reports is done at query time
  in Phase 3 (tenant_id on the report itself).

### Authentication Flow Changes

- **D-10:** `POST /auth/register` тАФ drop `publicTenant` middleware. No `X-Tenant-ID` header needed
  or accepted. Registration creates a global user account (citizen by default). Email is unique
  globally.

- **D-11:** `POST /auth/login` тАФ drop `publicTenant` middleware. Email lookup is global
  (`User.query().where('email', email)`). No tenant scope on the query.

- **D-12:** OAuth routes (`/auth/google/*`) тАФ also drop `publicTenant` middleware. OAuth user
  lookup/creation is global.

### Password Hashing

- **D-13:** Switch password hashing from **scrypt** to **argon2**. Install `argon2` npm package.
  Update `config/hash.ts` to use `argon2` as the default driver with recommended params.
  Update `User.ts` `withAuthFinder` call to `hash.use('argon2')`. Argon2 includes a random salt
  automatically тАФ no manual salting needed.

### Documentation Requirement

- **D-14:** Every file that implements the authorization hierarchy (TenantMiddleware, policies,
  middleware guards) MUST include a comment block explaining the three-tier auth model:
  superuser тЖТ membership тЖТ citizen. The implicit "presence in tenant_memberships = manager"
  logic is not obvious from reading code alone and must be made explicit.

### Claude's Discretion

- `TenantMembership` model: implement as a Lucid model with `belongsTo` relations to both `User`
  and `Tenant`. Naming, column order, and index strategy are Claude's call.
- Error message for 403 on missing membership: Claude decides the exact wording (must follow D-19
  from Phase 02 CONTEXT тАФ generic messages, no internal state exposure).
- Whether to add a `ctx.membership` typed property to the HttpContext contract or use a different
  mechanism to pass membership context to controllers/policies тАФ Claude decides.

</decisions>

## Existing Code Insights

### Reusable Assets
- `TenantMiddleware` (`app/shared/middleware/tenant_middleware.ts`): rewrite the inner logic to
  load `tenant_memberships` instead of `user.tenantId`. The transaction + set_config pattern stays.
- `PublicTenantMiddleware` (`app/shared/middleware/public_tenant_middleware.ts`): will be removed
  from register/login/oauth routes. May still be needed for truly public routes in Phase 3+.
- `HttpContext` contract (`app/shared/contracts/http_context.ts`): add `membership` and update
  `tenantId` тАФ currently typed as `string`, still valid.

### Established Patterns
- All migrations follow the pattern: `schema.createTable` тЖТ RLS тЖТ `GRANT`. Follow this for
  `tenant_memberships`.
- Feature-based folder structure: `TenantMembership` model goes in `app/models/` (shared across
  features, like `User` and `OAuthIdentity`).
- `withAuthFinder` mixin wires the hash driver. Changing to argon2 is a one-line change there.

### Integration Points
- Phase 02.2 will add admin endpoints that INSERT into `tenant_memberships`. The table and model
  must be production-ready from this phase.
- Phase 03 report submission will use `SELECT id FROM tenants WHERE ST_Contains(boundary, ST_Point(lng, lat))`.
  The `boundary` column must be in place.
- All future authenticated manager routes will use `middleware.tenant()` which will now validate
  memberships. The middleware contract changes are transparent to route files.

<specifics>
## Specific Ideas

- The user explicitly wants code comments explaining "presence in tenant_memberships = manager"
  because it's not obvious from the code alone. This is a non-negotiable documentation requirement
  for every file in the authorization path.
- "Edit original migrations" тАФ the user's explicit preference. Do NOT create new migrations to
  "fix" the schema. Edit `002_auth_users.ts` and `001_foundation_tenants.ts` in-place.
- Argon2 with automatic salt тАФ user's explicit preference over scrypt. Argon2 is the modern
  standard for password hashing (winner of Password Hashing Competition 2015). AdonisJS hash
  module handles salting automatically.

</specifics>

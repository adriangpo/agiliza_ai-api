## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** `users` table becomes global — remove `tenant_id` FK, remove `role` column. Add `is_superuser: boolean NOT NULL DEFAULT false`. Edit `002_auth_users.ts` in-place.
- **D-02:** Create `tenant_memberships` table: `id bigint serial PK`, `user_id bigint FK → users.id CASCADE`, `tenant_id uuid FK → tenants.id CASCADE`, `granted_by bigint FK → users.id`, `granted_at timestamp NOT NULL`, `UNIQUE(user_id, tenant_id)`. No `role` column — presence = manager.
- **D-03:** Remove RLS from `users` table entirely. Users are globally accessible by the app role.
- **D-04:** Add `boundary geography(POLYGON, 4326)` nullable column to `tenants` table. Edit `001_foundation_tenants.ts` in-place.
- **D-05:** `is_superuser boolean NOT NULL DEFAULT false` on `users`. Bouncer checks `user.isSuperuser` first before any tenant restriction.
- **D-06:** All members in `tenant_memberships` are equal — no permission tiers in v1.0.
- **D-07:** Token abilities are neutral at login time — all tokens carry `['reports:write', 'reports:read', 'profile:manage']`. Manager capabilities come from loading the `tenant_memberships` row when `X-Tenant-ID` is present. Bouncer reads `ctx.membership`.

- **D-09:** Citizens making authenticated requests do NOT use `TenantMiddleware`. Citizen routes have no tenant scoping at the auth layer.
- **D-10:** `POST /auth/register` — drop `publicTenant` middleware. No `X-Tenant-ID` needed.
- **D-11:** `POST /auth/login` — drop `publicTenant` middleware. Email lookup is global.
- **D-12:** OAuth routes — also drop `publicTenant` middleware.
- **D-13:** Switch password hashing from scrypt to argon2. Install `argon2` npm package. Update `config/hash.ts`. Update `withAuthFinder` call to `hash.use('argon2')`.
- **D-14:** Every file in the authorization path MUST include a comment block explaining the three-tier auth model: superuser → membership → citizen.

## Summary

This phase reworks the user-tenant relationship from 1:1 to M:N and introduces a three-tier authorization model (superuser → membership → citizen). The core changes are: making users global (no `tenant_id` on `users`), creating a `tenant_memberships` join table, rewriting `TenantMiddleware` to validate membership rather than read from the user record, updating auth routes to drop the public tenant middleware, switching password hashing from scrypt to argon2, and adding a PostGIS `boundary` column to `tenants`.

The phase edits two existing migrations in-place (dev environment, no production data). The test suite must be updated to reflect: no `X-Tenant-ID` header on register/login, global email uniqueness, and the new `TenantMiddleware` behavior. Cross-tenant leakage tests change from "same email in different tenants" to "user with no membership cannot access a tenant's data."

**Primary recommendation:** Follow the established migration/RLS patterns in the codebase exactly. The key insight is that RLS on the `users` table is being REMOVED (users are now global), while a new `tenant_memberships` table without RLS is added. The `TenantMiddleware` becomes a membership validator, not just a `set_config` caller.

### Pattern 1: `tenant_memberships` Migration with Existing Conventions

All migrations in this codebase follow: `createTable` → optional RLS → `GRANT`. The `tenant_memberships` table does NOT get RLS (it is used to determine what tenant context to apply, so it must be readable globally by the `app` role).

```typescript
// database/migrations/005_tenant_memberships.ts
// Source: established pattern in 001_foundation_tenants.ts and 003_auth_access_tokens.ts
this.schema.createTable('tenant_memberships', (table) => {
  table.bigIncrements('id').primary()
  table
    .bigInteger('user_id')
    .unsigned()
    .notNullable()
    .references('id')
    .inTable('users')
    .onDelete('CASCADE')
  table
    .uuid('tenant_id')
    .notNullable()
    .references('id')
    .inTable('tenants')
    .onDelete('CASCADE')
  table
    .bigInteger('granted_by')
    .unsigned()
    .notNullable()
    .references('id')
    .inTable('users')
  table.timestamp('granted_at').notNullable()
  table.unique(['user_id', 'tenant_id'])
  table.index(['user_id'])
  table.index(['tenant_id'])
})
// No RLS — membership table must be readable to determine tenant context
this.schema.raw(`GRANT SELECT, INSERT, UPDATE, DELETE ON tenant_memberships TO app`)
this.schema.raw(`GRANT USAGE, SELECT ON SEQUENCE tenant_memberships_id_seq TO app`)
```

### Pattern 2: PostGIS `geography` Column in Lucid Migrations

Knex (and therefore Lucid) has no native `geography()` column type. The only correct approach is `this.schema.raw()` with DDL. This is consistent with how this project already uses `rawQuery` for RLS policies.

```typescript
// database/migrations/001_foundation_tenants.ts — EDIT IN-PLACE
// Source: PostGIS official docs + knex raw pattern established in this codebase

// Add after table creation (or inline the ALTER if editing the createTable):
// Option A — ALTER TABLE (preferred when editing in-place):
this.schema.raw(`
  ALTER TABLE tenants
  ADD COLUMN IF NOT EXISTS boundary geography(POLYGON, 4326)
`)
// Option B — inside createTable using table.specificType() (knex API):
table.specificType('boundary', 'geography(POLYGON, 4326)').nullable()
```

**Recommendation:** Use `table.specificType('boundary', 'geography(POLYGON, 4326)').nullable()` inside the `createTable` block. This is the cleanest in-place edit and requires no separate `ALTER TABLE`. `specificType` is a Knex API that emits the column type verbatim — it is well-tested and documented.

For the `down()` function add `table.dropColumn('boundary')` inside a `this.schema.alterTable('tenants', ...)` call.

### Pattern 4: `TenantMembership` Lucid Model

The model should be a first-class Lucid model (not just a query builder call) because Phase 02.2 will need it for CRUD and the `TenantMiddleware` needs type-safe access to `tenantId`.

```typescript
// app/models/tenant_membership.ts
// Source: established User model pattern in this codebase
import { BaseModel, column, belongsTo } from '@adonisjs/lucid/orm'
import type { BelongsTo } from '@adonisjs/lucid/types/relations'
import { DateTime } from 'luxon'
import User from '#models/user'
import Tenant from '#models/tenant'

export default class TenantMembership extends BaseModel {
  static table = 'tenant_memberships'

  @column({ isPrimary: true })
  declare id: number

  @column()
  declare userId: number

  @column()
  declare tenantId: string

  @column()
  declare grantedBy: number

  @column.dateTime()
  declare grantedAt: DateTime

  @belongsTo(() => User)
  declare user: BelongsTo<typeof User>

  @belongsTo(() => User, { foreignKey: 'grantedBy' })
  declare grantedByUser: BelongsTo<typeof User>

  @belongsTo(() => Tenant)
  declare tenant: BelongsTo<typeof Tenant>
}
```

**Note on `Tenant` model:** The project does not currently have a `Tenant` Lucid model (tenants are queried via raw `db.from('tenants')`). This phase may need to create a minimal `Tenant` model, or skip the `belongsTo(() => Tenant)` relation and use raw queries in the middleware. The simpler path is raw query in `TenantMiddleware`:

```typescript
const membership = await db
  .from('tenant_memberships')
  .where('user_id', user.id)
  .where('tenant_id', tenantId)
  .first()
```

### Pattern 5: Rewritten `TenantMiddleware`

The existing `TenantMiddleware` reads `user.tenantId`. The rewritten version must:
1. Read `X-Tenant-ID` header (not from the user record)
2. Check `user.isSuperuser` (bypass for platform admins)
3. Query `tenant_memberships` for the user+tenant pair
4. Set `app.tenant_id`, `ctx.tenantId`, `ctx.membership` (or `ctx.db`)
5. Return 403 if no membership and not superuser

```typescript
// app/shared/middleware/tenant_middleware.ts — REWRITE
// Source: existing TenantMiddleware pattern + D-08 decisions
//
// ┌─────────────────────────────────────────────────────────────────┐
// │ THREE-TIER AUTHORIZATION MODEL (D-14 — MUST be in every file)  │
// │ 1. user.isSuperuser === true  → platform admin, bypasses all   │
// │    tenant restrictions                                          │
// │ 2. tenant_memberships row exists for (user.id, tenantId)       │
// │    → manager for that tenant (presence = manager role)         │
// │ 3. Neither → citizen, no tenant context allowed                │
// └─────────────────────────────────────────────────────────────────┘

export default class TenantMiddleware {
  async handle(ctx: HttpContext, next: NextFn) {
    const user = ctx.auth.user!
    const tenantId = ctx.request.header('X-Tenant-ID')

    if (!tenantId) {
      return ctx.response.forbidden({ message: 'Tenant context required.' })
    }

    // Superuser bypass — platform admins have access to all tenants
    let membership: TenantMembership | null = null
    if (!user.isSuperuser) {
      membership = await db
        .from('tenant_memberships')
        .where('user_id', user.id)
        .where('tenant_id', tenantId)
        .first()

      if (!membership) {
        return ctx.response.forbidden({ message: 'Access denied.' })
      }
    }

    await db.transaction(async (trx) => {
      await trx.rawQuery(`SELECT set_config('app.tenant_id', ?, ?)`, [tenantId, 'true'])
      ctx.tenantId = tenantId
      ctx.membership = membership  // null for superusers
      ctx.db = trx
      await next()
    })
  }
}
```

### Pattern 6: `HttpContext` Augmentation for `ctx.membership`

The existing contract file at `app/shared/contracts/http_context.ts` is the right place to add `ctx.membership`:

```typescript
// app/shared/contracts/http_context.ts
import type { TransactionClientContract } from '@adonisjs/lucid/types/database'
import type TenantMembership from '#models/tenant_membership'
import '@adonisjs/core/http'

declare module '@adonisjs/core/http' {
  interface HttpContext {
    tenantId: string
    db: TransactionClientContract
    membership: TenantMembership | null  // null for superusers, TenantMembership for managers
  }
}
```

**Important:** `membership` should be typed as `TenantMembership | null` (not optional `?`) because the middleware always sets it. Policies that need to check manager access should use `ctx.membership !== null || ctx.auth.user!.isSuperuser`.

### Pattern 7: Argon2 Hash Configuration

The `argon2` npm package is a peer dependency of the AdonisJS hash driver — it must be installed separately. The `@adonisjs/core` hash module lazily imports it.

```typescript
// config/hash.ts — UPDATE
import { defineConfig, drivers } from '@adonisjs/core/hash'

const hashConfig = defineConfig({
  default: 'argon2',
  list: {
    argon2: drivers.argon2({
      variant: 'id',  // argon2id — recommended, balances security and side-channel resistance
      iterations: 3,
      memory: 65536,  // 64 MiB — OWASP minimum recommendation
      parallelism: 4,
      saltSize: 16,   // auto-generated, 16 bytes (128-bit) default
    }),
  },
})
```

`ArgonConfig` fields (verified from `@adonisjs/hash` type definitions):
- `variant?: 'id' | 'i' | 'd'` — default `'id'` (argon2id is the recommended variant)
- `version?: 0x10 | 0x13` — default `0x13` (version 19, latest)
- `iterations?: number` — default `3`
- `memory?: number` — in KiB, default `65536` (64 MiB)
- `parallelism?: number` — default `4`
- `saltSize?: number` — in bytes, default `16`
- `hashLength?: number` — in bytes, default `32`

The `withAuthFinder` call in `User` model changes from:
```typescript
const AuthFinder = withAuthFinder(() => hash.use('scrypt'), { ... })
```
to:
```typescript
const AuthFinder = withAuthFinder(() => hash.use('argon2'), { ... })
```

**Hash migration concern:** Existing test data uses scrypt hashes. Because this is a dev environment with no production data, running `make migrate-fresh` re-seeds from scratch. No hash migration is needed for existing hashes. The `withAuthFinder` mixin also includes a `needsRehash()` mechanism — this is a future concern, not for this phase.

### Pattern 8: Lucid `manyToMany` Decorator (Reference Only)

While the primary access pattern for this phase uses direct queries against `tenant_memberships`, the `User` model CAN expose a `manyToMany` relation for future convenience. The `manyToMany` decorator options (verified from Lucid source):

```typescript
// app/models/user.ts — optional addition
@manyToMany(() => Tenant, {
  pivotTable: 'tenant_memberships',
  pivotColumns: ['granted_by', 'granted_at'],
  pivotForeignKey: 'user_id',
  pivotRelatedForeignKey: 'tenant_id',
})
declare tenants: ManyToMany<typeof Tenant>
```

**However:** This requires a `Tenant` Lucid model. Given the middleware only needs a direct query for the membership check, the simpler path is a raw DB query in `TenantMiddleware` and a full `TenantMembership` model for Phase 02.2's admin endpoints. Do not create a `Tenant` Lucid model just for this relation if it adds no value in Phase 02.1.

### Pattern 9: OAuth Identities Migration — Remove `tenant_id`

`004_auth_oauth_identities.ts` currently has a `tenant_id` FK and a unique constraint on `(tenant_id, provider, provider_user_id)`. With global users, this should become unique on `(provider, provider_user_id)`. The `tenant_id` column must be removed.

The `tenant_id` index `table.index(['tenant_id'])` should also be removed.

### Anti-Patterns to Avoid

- **Using `db.rawQuery()` instead of `this.schema.raw()` inside migrations for schema changes:** `db.rawQuery()` is immediate; `this.schema.raw()` is deferred and runs after `createTable`. The codebase already flags this in comments.
- **Adding RLS to `tenant_memberships`:** This table determines tenant context — if RLS filtered it, the middleware could never load a membership. No RLS on this table.
- **Adding RLS back to `users`:** D-03 explicitly removes RLS from `users`. With global users, there is no `tenant_id` to enforce. Do not add any policy.
- **Using `table.integer()` instead of `table.bigInteger()` for `user_id` and `granted_by` FKs:** `users.id` is `bigIncrements` (bigint). Using plain `integer` would silently truncate large user IDs. This pattern is already documented in `003_auth_access_tokens.ts`.
- **Making `ctx.membership` optional (`?`) rather than typed as `T | null`:** Optional would allow TypeScript to silently skip null checks. Explicit `| null` forces explicit checks at call sites.
- **Calling `next()` outside the `db.transaction()` block in `TenantMiddleware`:** The RLS `set_config` is transaction-scoped. If `next()` runs outside the transaction, the RLS setting has already reset to null and all queries will return no data.
- **Removing `publicTenant` from route files without understanding what replaces it:** Register/login routes no longer need tenant middleware at all. Authenticated routes with tenant context need the rewritten `middleware.tenant()`. Do not accidentally wire register/login to the new `middleware.tenant()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Argon2 hashing with automatic salt | Custom hash function | `argon2` npm + `drivers.argon2()` in `config/hash.ts` | Salt management, PHC format storage, rehash detection built-in |
| PostGIS column type in migration | String type + app-layer serialization | `table.specificType('boundary', 'geography(POLYGON, 4326)')` | Geography type enables `ST_Contains`, `ST_DWithin` in Phase 3 queries |
| Membership access check in controller | Inline DB query per controller | `TenantMiddleware` validates once per request, attaches `ctx.membership` | Consistent enforcement — no route can bypass membership check |
| HttpContext extension | Thread-local storage or request-scoped service | TypeScript module augmentation in `app/shared/contracts/http_context.ts` | AdonisJS recommended pattern; type-safe at call sites |

## Common Pitfalls

### Pitfall 1: Existing Tests Still Send `X-Tenant-ID` to `/auth/register` and `/auth/login`

**What goes wrong:** After removing `publicTenant` middleware from register/login routes, tests that send `X-Tenant-ID: tenantId` will still pass (the header is ignored, not rejected). However, tests that TEST the header (e.g., asserting that two tenants can have the same email) will break because email is now globally unique.

**Why it happens:** The migration removes `UNIQUE(tenant_id, email)` and adds `UNIQUE(email)` globally. Tests that register the same email in two different tenants will now get a 409 conflict.

**How to avoid:** Update `cross_tenant.spec.ts` — the "same email in different tenants" test is now invalid. Replace it with "user with no membership to tenant A gets 403 when using tenant middleware."

**Warning signs:** `409 Conflict` in the test that was previously testing cross-tenant email uniqueness.

### Pitfall 2: `TenantMiddleware` Tests Use Mocked `ctx.auth.user` Without `isSuperuser`

**What goes wrong:** The existing `tenant_middleware.spec.ts` mocks `ctx.auth.user` as `{ tenantId }`. The rewritten middleware now reads `user.isSuperuser` and queries `tenant_memberships`. The mock is missing both.

**Why it happens:** The test predates the M:N architecture. The mock doesn't match the new User model shape.

**How to avoid:** Update `tenant_middleware.spec.ts` to mock `{ id: someId, isSuperuser: false }` and insert a row into `tenant_memberships` in the test setup.

### Pitfall 3: `oauth_identities` Unique Constraint Becomes Global After Removing `tenant_id`

**What goes wrong:** The current unique constraint is `(tenant_id, provider, provider_user_id)`. After removing `tenant_id`, if the constraint is not updated to `(provider, provider_user_id)`, the same Google account could be linked to multiple users globally, which is incorrect.

**Why it happens:** Editing the migration in-place — easy to forget to update the unique constraint alongside the column removal.

**How to avoid:** In `004_auth_oauth_identities.ts`, drop `table.unique(['tenant_id', 'provider', 'provider_user_id'])` and replace with `table.unique(['provider', 'provider_user_id'])`. Also remove `table.index(['tenant_id'])` and the `tenant_id` column itself.

### Pitfall 4: Auth Service Still Scopes `User.query()` by Tenant

**What goes wrong:** `auth_service.ts` calls `User.query().where('email', email).first()`. After removing `tenant_id` from `users`, this query still works BUT the existing `TenantMiddleware` may have set `app.tenant_id` in the DB session, and if the `users` table still has RLS (it shouldn't after D-03), the query returns no results.

**Why it happens:** Running migrate-fresh while forgetting to also remove the RLS policies from the migration file. Editing the migration in-place means the RLS `ALTER TABLE` statements must be deleted, not just commented out.

**How to avoid:** In `002_auth_users.ts`, delete the three `this.schema.raw()` calls that enable RLS, FORCE RLS, and create the tenant isolation policy. The table must have no RLS policy after this phase.

### Pitfall 5: `AuthService.createToken()` Still Uses `user.role` for Ability Selection

**What goes wrong:** `auth_service.ts` currently checks `user.role === 'manager'` to decide token abilities. After removing `role` from `users`, this crashes with a runtime error.

**Why it happens:** The `role` column is removed from the model and the DB, but the service code still references it.

**How to avoid:** Update `AuthService.createToken()` to always issue neutral abilities per D-07: `['reports:write', 'reports:read', 'profile:manage']`. Manager capabilities are now determined at request time via `ctx.membership`, not at token issuance time.

### Pitfall 6: `withAuthFinder` Changed to `hash.use('argon2')` but `argon2` Not Installed

**What goes wrong:** `hash.use('argon2')` triggers lazy loading of the `argon2` npm package. If it's not installed, the first login or register request throws `Cannot find package 'argon2'` at runtime (not at startup).

**Why it happens:** `@adonisjs/core` bundles the driver configuration but not the underlying crypto library itself.

**How to avoid:** Run `pnpm add argon2` before changing `config/hash.ts`. Verify with `node -e "require('argon2')"`.

### Pitfall 7: `AuthController.register()` Still Creates User with `tenantId` and `role`

**What goes wrong:** After removing `tenant_id` and `role` from the `users` table migration, `AuthController.register()` still tries to insert `{ tenantId, role: 'citizen' }` into the user record. This causes a database error (unknown column).

**Why it happens:** Register/login are tightly coupled to the old schema. Multiple call sites must change atomically with the migration.

**How to avoid:** Update the controller, service, and model in the same plan that edits the migration. Never edit the migration without also updating the application code that writes to that table.

## Open Questions

1. **Circular import risk: `TenantMembership` → `User` → `TenantMembership`**
   - What we know: `User` will have `hasMany(() => TenantMembership)` (optional), and `TenantMembership` will have `belongsTo(() => User)`. Lucid models use lazy arrow functions for relations, which breaks circular module dependency at import time.
   - What's unclear: Whether adding `hasMany(() => TenantMembership)` on `User` is needed in this phase at all (Phase 02.2 will use it more). Omitting it from `User` in Phase 02.1 avoids any potential issue.
   - Recommendation: Do NOT add the `hasMany` relation on `User` in this phase. The `TenantMembership` model's `belongsTo(() => User)` is sufficient for Phase 02.2's use cases.

2. **`oauth_identities.tenant_id` removal: does Phase 02 code use this column?**
   - What we know: `SocialAuthController` uses `tenant_id` to scope OAuth identity lookup. With global users, OAuth lookups should be global too.
   - What's unclear: The exact queries in `social_auth_controller.ts` and `oauth_identity.ts` that reference `tenant_id`.
   - Recommendation: Read `social_auth_controller.ts` and `OAuthIdentity` model before implementing, and update all `tenant_id` references there.

3. **Seeder data for tests: how to create a superuser and grant memberships?**
   - What we know: There's no admin endpoint in this phase. Phase 02.2 adds admin endpoints.
   - What's unclear: Do tests in this phase need a superuser or granted membership to test the new `TenantMiddleware`?
   - Recommendation: Test setup should insert rows directly into `tenant_memberships` via `db.table('tenant_memberships').insert(...)` (same pattern as tenant insertion in existing tests). No seeder changes needed.

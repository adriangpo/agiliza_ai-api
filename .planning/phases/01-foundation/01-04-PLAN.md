    ```sql
    -- database/setup/create_roles.sql
    -- INFRA-04: Two DB roles.
    -- Run as superuser (postgres) ONCE during initial environment setup.
    -- migrator: owns tables, runs DDL, sets up RLS policies.
    --           Subject to FORCE ROW LEVEL SECURITY on tenant tables.
    -- app: DML only (SELECT, INSERT, UPDATE, DELETE). RLS-restricted.
    --      Used by the running application. Never owns tables. Never runs DDL.

    -- Create roles (idempotent — skip if already exists)
    DO $$
    BEGIN
      IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'migrator') THEN
        CREATE ROLE migrator LOGIN PASSWORD 'migrator_password' NOINHERIT;
      END IF;
      IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'app') THEN
        CREATE ROLE app LOGIN PASSWORD 'app_password' NOINHERIT;
      END IF;
    END $$;

    -- Grant migrator schema creation rights on the database
    GRANT CREATE ON DATABASE agiliza_ai TO migrator;
    GRANT CREATE ON DATABASE agiliza_ai_test TO migrator;

    -- Grant app role connect access only (DML grants happen inside each migration,
    -- after the table is created, scoped to the specific table)
    GRANT CONNECT ON DATABASE agiliza_ai TO app;
    GRANT CONNECT ON DATABASE agiliza_ai_test TO app;

    -- Note: After each migration creates a table, the migration itself grants:
    -- GRANT SELECT, INSERT, UPDATE, DELETE ON <table> TO app;
    -- GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO app;
    -- This keeps grants minimal and explicit per table (D-07: app is DML only).
    ```

    // Runs as 'migrator' role (DDL owner).

    export default class extends BaseSchema {
      async up() {
        // PostGIS for geospatial queries (ST_DWithin, ST_MakePoint, etc.)
        await this.db.rawQuery('CREATE EXTENSION IF NOT EXISTS postgis')

      async down() {
        // Extensions are intentionally NOT dropped — they may be needed by other tables.
        // Dropping postgis in down() would cascade-drop all geography columns.
      }
    }

    Step 2 — Create `database/migrations/001_foundation_tenants.ts`:
    Creates the `tenants` table with UUID v7 PK (application-generated via `uuidv7` package).
    Tenants is NOT a tenant-scoped table (it IS the tenant registry). No RLS on tenants itself.
    The migration also includes the REUSABLE RLS macro that all future tenant-scoped migrations will call.

    ```typescript
    // database/migrations/001_foundation_tenants.ts
    // INFRA-05b: UUID v7 PK for tenants. D-09.
    // INFRA-04/05: Establishes the RLS grant pattern for future tables.
    // Runs as 'migrator' role (DDL owner).
    import { BaseSchema } from '@adonisjs/lucid/schema'

    export default class extends BaseSchema {
      async up() {
        // Tenants table — the top-level tenant registry.
        // UUID v7 PK: generated by application layer using uuidv7 npm package.
        // NOT tenant-scoped itself — no RLS needed here.
        this.schema.createTable('tenants', (table) => {
          table.uuid('id').primary().comment('UUID v7 — generated application-side via uuidv7 package')
          table.string('name', 255).notNullable()
          table.string('slug', 100).notNullable().unique()
          table.boolean('active').notNullable().defaultTo(true)
          table.timestamps(true, true)
        })

        // Grant app role access to tenants (read-only: app never inserts tenants)
        await this.db.rawQuery(`GRANT SELECT ON tenants TO app`)

        // ──────────────────────────────────────────────────────────────────────
        // CANONICAL RLS PATTERN FOR ALL FUTURE TENANT-SCOPED TABLES
        // Copy this block into every migration that creates a tenant-scoped table.
        // Replace 'some_table' with the actual table name.
        //
        // await this.db.rawQuery(`
        //   -- 1. Enable RLS
        //   ALTER TABLE some_table ENABLE ROW LEVEL SECURITY;
        //
        //   -- 2. FORCE RLS applies RLS even to the table owner (migrator role)
        //   --    This prevents accidental bypass via direct DB connections. INFRA-05.
        //   ALTER TABLE some_table FORCE ROW LEVEL SECURITY;
        //
        //   -- 3. Create tenant isolation policy
        //   --    current_setting('app.tenant_id', true) — the 'true' arg returns NULL
        //   --    instead of error when setting is not set (safe outside transactions)
        //   CREATE POLICY tenant_isolation ON some_table
        //     USING (tenant_id = current_setting('app.tenant_id', true)::uuid)
        //     WITH CHECK (tenant_id = current_setting('app.tenant_id', true)::uuid);
        //
        //   -- 4. Grant DML to app role (SELECT, INSERT, UPDATE, DELETE — no DDL)
        //   GRANT SELECT, INSERT, UPDATE, DELETE ON some_table TO app;
        //   GRANT USAGE, SELECT ON some_table_id_seq TO app;
        // `)
        // ──────────────────────────────────────────────────────────────────────
      }

      async down() {
        this.schema.dropTableIfExists('tenants')
      }
    }
    ```

    CRITICAL notes on the RLS policy:
    - Use `current_setting('app.tenant_id', true)` NOT `current_setting('app.tenant_id')`. The second argument `true` means "return NULL if missing" instead of raising an error. This is required for the first request before TenantMiddleware sets the value.
    - Use `FORCE ROW LEVEL SECURITY` (not just `ENABLE ROW LEVEL SECURITY`). FORCE applies RLS even to the table owner (migrator). Without FORCE, migrator bypasses all policies — a dangerous loophole. (D-08)
    - The `WITH CHECK` clause enforces RLS on INSERT and UPDATE, not just SELECT. Omitting it allows cross-tenant writes.
